Java · Servlets · PostgreSQL · Jackson · Bruno

Le contexte.

SAÉ S4.A02.1 IUT de Lille, projet en binôme avec Sewan Hamida. Objectif : concevoir une API REST Java complète pour gérer la collecte de déchets dans une commune — utilisateurs, points de collecte, types de déchets acceptés, dépôts effectués, et leaderboard des recycleurs les plus actifs.

Contraintes fortes : validations métier strictes (poids positif, capacité non dépassée), négociation de contenu (JSON ou XML selon le header Accept), tests d'API exhaustifs.

La stack.

LangageJava

SérialisationJackson (JSON + XML)
TestsBruno (20+ scénarios)
ImportCSV batch

L'architecture.

Pattern DAO/DTO avec singletons pour les DataSources, requêtes préparées (protection contre injection SQL). 6 DTOs (WasteType, CollectionPoint, Deposit, User, UserScore, PointStatus) et 5 DAOs. 4 servlets exposant chacune une ressource REST avec verbes HTTP standards.

Deux vues SQL calculent les agrégats côté base : UserLeaderboard (score total par utilisateur via SUM(poids * pointsPerKilo)) et PointRemplissage (taux de remplissage et flag full).

Les défis.

  • Implémenter la négociation de contenu via le header Accept (JSON par défaut, XML sur demande). Solution : utilitaire de sérialisation Jackson qui choisit le mapper selon le header reçu.
  • Valider chaque création avec les bons codes HTTP (400 poids négatif, 403 point plein, 404 ressource introuvable, 409 login déjà pris). Solution : helper de validation centralisé.
  • Importer des dépôts en masse depuis un CSV avec rapport détaillé. Solution : programme ImportDeposits.java qui parse, valide ligne par ligne, et produit un récap succès/erreurs.

Les apprentissages.

Servlets Java avec annotations @WebServlet, pattern DAO/DTO, requêtes préparées contre l'injection SQL, sérialisation Jackson en JSON et XML, vues SQL pour calculs métier, écriture de 20+ tests d'API avec Bruno, et la valeur d'une API stateless bien typée pour un frontend ou mobile.
